# AES-256-GCM for Vault Encryption

Latch uses AES-256-GCM (Galois/Counter Mode) for encrypting the Vault. We chose this over XChaCha20-Poly1305 because AES-GCM benefits from hardware acceleration (AES-NI on x86 processors), which matters for the encrypt/decrypt path triggered on every Vault save and Access. XChaCha20-Poly1305 has better nonce-misuse resistance but is software-only, making it slower on common desktop hardware.

## Considered Options

- **XChaCha20-Poly1305**: Better nonce-misuse resistance, no hardware dependency. Rejected because AES-NI gives measurable speed wins on x86 desktops (Latch's primary platform), and we generate fresh random nonces via `OsRng` for every encryption — nonce reuse is not a concern.

- **ChaCha20-Poly1305 (standard)**: 96-bit nonce carries collision risk under the birthday bound. Rejected in favor of AES-GCM's similar nonce size but with hardware acceleration.
